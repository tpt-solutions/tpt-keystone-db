# Harbor Phase 15: Replace stubs with proper implementations

## Overview

Replace all 10 `unimplemented_source!` stubs in `tpt-harbor/src/sources/mod.rs` with real `SourceConnector` implementations. Each connector gets its own module file with a hand-written wire protocol client (or HTTP client for REST-based sources), schema discovery, cursor-batched snapshot reads, and row checksum computation. CDC replication is implemented where the source natively supports change feeds; otherwise a poll-based re-scan is the scope-cut fallback.

## Architecture pattern

Every connector follows the same shape established by `sources/postgres.rs`:

1. A `<Name>Source` struct holding connection state
2. An `async fn connect(...)` constructor
3. `impl SourceConnector` with `discover()`, `snapshot_table()`, `replicate()`, `row_checksums()`
4. Type mapping via new functions in `schema.rs` (e.g. `from_mysql_type`, `from_mongodb_type`)
5. Wire protocol client either in its own file (complex binary protocols) or inline (simple HTTP)

## Connectors to implement (one per file in `tpt-harbor/src/sources/`)

### 1. Harbor/GIS — `sources/postgis.rs` (PostGIS → Meridian)
- **Wire protocol**: Reuse existing `pgwire.rs` — PostGIS speaks Postgres wire protocol
- **Type mapping**: Add `from_postgis_type()` to `schema.rs`; PostGIS geometry/geography/point/line/polygon → Keystone `GEOMETRY` (WKT text). Query `SELECT * FROM geometry_columns` for table discovery, or detect geometry columns via `pg_type.typname IN ('geometry','geography')` during information_schema discovery
- **Snapshot**: Same cursor-batched SELECT as `PostgresSource`, but wrap geometry columns with `ST_AsText(col)` so PostGIS WKB comes back as WKT that Keystone can store directly
- **CDC**: Same `pgoutput` logical replication as PG — geometry columns will arrive as hex WKB; decode to WKT in the CDC event handler using hand-written WKB-to-WKT conversion (or `ST_AsText` wrapper for the live stream)
- **Row checksums**: Same approach as PG
- **File**: `tpt-harbor/src/sources/postgis.rs`

### 2. Harbor/MySQL — `sources/mysql.rs` (MySQL/MariaDB → Keystone)
- **Wire protocol**: Hand-written MySQL client/server protocol over TCP (port 3306). Handshake (server greeting → auth response), COM_QUERY, result set decoding (column definitions + row data + EOF markers)
- **Discovery**: `SHOW DATABASES` + `information_schema.tables` + `information_schema.columns` (MySQL has information_schema, same concept as PG)
- **Snapshot**: `SELECT * FROM table` with streaming results
- **CDC**: Scope cut — return `Unimplemented` for `replicate()` initially, with a TODO for binlog CDC. For now the migration path is snapshot-only
- **Row checksums**: `SELECT * FROM table ORDER BY pk` + hash
- **Type mapping**: Add `from_mysql_type()` — TINYINT/SMALLINT/INT/BIGINT/FLOAT/DOUBLE/DECIMAL/VARCHAR/TEXT/JSON/DATETIME/TIMESTAMP/DATE/BLOB → Keystone types
- **File**: `tpt-harbor/src/sources/mysql.rs`

### 3. Harbor/MSSQL — `sources/mssql.rs` (SQL Server → Keystone)
- **Wire protocol**: Hand-written TDS 7.4 protocol over TCP (port 1433). Login packet, pre-login handshake, COM_QUERY, token-based result set decoding (COLMETADATA + ROW tokens)
- **Discovery**: `SELECT * FROM information_schema.tables` + `information_schema.columns` (SQL Server has information_schema)
- **Snapshot**: Cursor-based `DECLARE c CURSOR FOR SELECT * FROM t FETCH NEXT` or direct `SELECT` with streaming
- **CDC**: Scope cut — return `Unimplemented` for `replicate()`. SQL Server CDC exists but is complex (reading `cdc.fn_cdc_get_all_changes_<capture_instance>`)
- **Row checksums**: Same approach as PG
- **Type mapping**: Add `from_mssql_type()` — tinyint/smallint/int/bigint/float/real/decimal/numeric/varchar/nvarchar/text/ntext/datetime/datetime2/binary/varbinary/json/bit → Keystone types
- **File**: `tpt-harbor/src/sources/mssql.rs`

### 4. Harbor/Mongo — `sources/mongodb.rs` (MongoDB → Canopy)
- **Wire protocol**: Hand-written MongoDB OP_MSG protocol over TCP (port 2717). Hello/OP_MSG handshake, `isMaster`/`hello`, OP_MSG command framing
- **Discovery**: `listCollections` command to enumerate collections, `listCollections` with schema info for field types. Each MongoDB collection → one Keystone table; `_id` field → primary key
- **Snapshot**: `find` command with batch size, iterate via `getMore`
- **CDC**: Scope cut — return `Unimplemented` for `replicate()`. Change streams/oplog tailing is complex; snapshot-only for initial implementation
- **Row checksums**: `find` all documents, serialize to canonical bytes, hash
- **Type mapping**: Add `from_mongodb_type()` — int32/int64/double/string/bool/date/objectId/array/document/binary/null → Keystone types. Flatten nested documents as JSONB
- **File**: `tpt-harbor/src/sources/mongodb.rs`

### 5. Harbor/Graph — `sources/neo4j.rs` (Neo4j → Plexus)
- **Wire protocol**: Hand-written Bolt protocol v4+ over TCP (port 7687). INIT handshake, RUN/PULL messages, record decoding
- **Discovery**: `CALL db.labels()` for node labels, `CALL db.relationshipTypes()` for relationship types, then `MATCH (n:Label) RETURN DISTINCT keys(n)` per label for properties
- **Snapshot**: `MATCH (n) RETURN n` per label, flatten node properties into rows
- **CDC**: Scope cut — Neo4j doesn't have a standard CDC mechanism; return `Unimplemented`
- **Row checksums**: Same approach
- **Type mapping**: Add `from_neo4j_type()` — Boolean/Integer/Float/String/List/Map/Date/DateTime/Duration → Keystone types. Properties map to columns
- **File**: `tpt-harbor/src/sources/neo4j.rs`

### 6. Harbor/TimeSeries — `sources/influxdb.rs` (InfluxDB → Chronos)
- **Wire protocol**: HTTP/1.1 over TCP. Hand-rolled minimal HTTP client (request builder + response parser on `tokio::net::TcpStream`). InfluxDB 1.x uses `/query` endpoint; InfluxDB 2.x uses `/api/v2/query`
- **Discovery**: `SHOW DATABASES` + `SHOW MEASUREMENTS` + `SHOW FIELD KEYS ON <db> FROM <measurement>`
- **Snapshot**: `SELECT * FROM <measurement>` via InfluxDB's line protocol or CSV response
- **CDC**: Scope cut — return `Unimplemented`
- **Row checksums**: Same approach
- **Type mapping**: Add `from_influx_type()` — float/integer/boolean/string/tag/time → Keystone types. Tags become TEXT columns, time becomes TIMESTAMP
- **File**: `tpt-harbor/src/sources/influxdb.rs`

### 7. Harbor/Stream — `sources/kafka.rs` (Kafka → Flux)
- **Wire protocol**: Hand-written Kafka binary protocol over TCP (port 9092). ApiVersions → Metadata → FindCoordinator → JoinGroup → Fetch
- **Discovery**: Metadata API to list topics and their partitions
- **Snapshot**: Consumer group Fetch to read all messages from each topic/partition
- **CDC**: Kafka IS a log — consume from earliest offset, which IS the CDC path. This is the one connector where `replicate()` is real: keep fetching new offsets
- **Row checksums**: Hash of all messages in a topic
- **Type mapping**: Kafka messages are opaque bytes; treat as BYTEA or TEXT. Topic metadata (key, value, offset, timestamp) maps to columns
- **File**: `tpt-harbor/src/sources/kafka.rs`

### 8. Harbor/Vector — `sources/vector.rs` (Pinecone/Weaviate/Qdrant → Prism)
- **Wire protocol**: HTTP/1.1 REST APIs. Same hand-rolled minimal HTTP client as InfluxDB
- **Pinecone**: `GET /vectors/list` + `POST /vectors/fetch` for discovery + snapshot
- **Weaviate**: `GET /v1/objects` + `GET /v1/schema` for discovery + snapshot  
- **Qdrant**: `GET /collections` + `POST /collections/{name}/points/scroll` for discovery + snapshot
- **CDC**: Scope cut — vector databases don't have standard CDC; return `Unimplemented`
- **Row checksums**: Same approach
- **Type mapping**: Vector IDs → primary key, vector data → BYTEA or JSONB, metadata → JSONB
- **File**: `tpt-harbor/src/sources/vector.rs`

### 9. Harbor/Search — `sources/elasticsearch.rs` (Elasticsearch → Canopy)
- **Wire protocol**: HTTP/1.1 REST API (port 9200). Same hand-rolled HTTP client
- **Discovery**: `GET /_cat/indices?format=json` + `GET /<index>/_mapping` for field types
- **Snapshot**: `POST /<index>/_search` with `scroll=1m` + `POST /_search/scroll` for pagination
- **CDC**: Scope cut — return `Unimplemented`. Could use the `_search` API with `sort` + `search_after` for polling, but that's deferred
- **Row checksums**: Same approach
- **Type mapping**: Add `from_es_type()` — text/keyword/long/integer/short/byte/double/float/boolean/date/object/nested → Keystone types. Nested objects flatten as JSONB
- **File**: `tpt-harbor/src/sources/elasticsearch.rs`

### 10. Harbor/Oracle — `sources/oracle.rs` (Oracle → Keystone)
- **Wire protocol**: Hand-written Oracle Net (TNS) protocol over TCP (port 1521). This is the most complex binary protocol in the set — TNS connect, TTC data types, cursor operations
- **Discovery**: Oracle data dictionary: `SELECT * FROM user_tables` + `SELECT * FROM user_tab_columns` + `SELECT * FROM user_constraints` + `SELECT * FROM user_cons_columns`
- **Snapshot**: `DECLARE CURSOR c FOR SELECT * FROM t FETCH NEXT` (Oracle syntax)
- **CDC**: Scope cut — LogMiner/GoldenGate is complex; return `Unimplemented`
- **Row checksums**: Same approach
- **Type mapping**: Add `from_oracle_type()` — NUMBER/VARCHAR2/CHAR/CLOB/DATE/TIMESTAMP/BLOB/BFILE/RAW/NVARCHAR2 → Keystone types
- **File**: `tpt-harbor/src/sources/oracle.rs`

## Shared HTTP client helper

For InfluxDB, Elasticsearch, Pinecone/Weaviate/Qdrant — create a minimal shared HTTP/1.1 client in `tpt-harbor/src/http.rs`:

```rust
/// Minimal HTTP/1.1 client over tokio TCP — enough for REST API sources.
pub struct HttpClient { stream: TcpStream }

impl HttpClient {
    pub async fn connect(addr: &str) -> Result<Self>;
    pub async fn get(&mut self, path: &str) -> Result<HttpResponse>;
    pub async fn post(&mut self, path: &str, body: &[u8]) -> Result<HttpResponse>;
}

pub struct HttpResponse {
    pub status: u16,
    pub headers: HashMap<String, String>,
    pub body: Vec<u8>,
}
```

This avoids duplicating HTTP parsing across 4+ connectors.

## Implementation order

1. `http.rs` (shared HTTP client) — needed by 4 connectors
2. `postgis.rs` — easiest (reuses pgwire.rs, just add type mapping + ST_AsText wrapping)
3. `mysql.rs` — well-documented protocol, relational DB (similar to PG pattern)
4. `mssql.rs` — TDS protocol, relational DB
5. `mongodb.rs` — OP_MSG + BSON
6. `neo4j.rs` — Bolt protocol
7. `influxdb.rs` — HTTP API (uses http.rs)
8. `kafka.rs` — binary protocol
9. `vector.rs` — HTTP REST (uses http.rs)
10. `elasticsearch.rs` — HTTP REST (uses http.rs)
11. `oracle.rs` — TNS protocol (most complex, last)
